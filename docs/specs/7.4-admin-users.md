## 1. Goal & non-goals

### Goal

Third screen-phase of v2. Migrates `src/views/AdminUsersView.vue`
(926 LOC, the largest view file) to consume the Phase 7.1 atom set:
BaseCard (bigstat), BaseTable, BaseBadge, BaseModal, BaseButton.

DESIGN-v2 §5 #3 names AdminUsersView as "most-used admin table",
the first heavy BaseTable consumer at scale — 8 columns, composite
user cell, paging, filter tabs.

### In scope

- Migrate `src/views/AdminUsersView.vue` only.
- **BaseCard (bigstat) ×4**: the four KPI tiles (Usuarios totales /
  Activos / Restringidos / Saldo bajo gestión).
- **BaseTable ×1**: the data-table (8 columns) becomes `<BaseTable>`
  with typed `Column<AdminUser>[]` and 5 `cell-<key>` slots
  (user composite, status, kyc, wallets-count, balance). Existing
  `row-click → openDrawer(u)` wires to BaseTable's `@row-click`
  event 1:1.
- **BaseBadge ×~3 per row**:
  - status-dot → `<BaseBadge :tone>` (soft variant, dot enabled):
    `active → success`, `banned → danger`, `deleted → neutral`.
  - kyc-badge → `<BaseBadge variant="outline" :tone>` with tone
    derived from KYC level (L0/L1 → warning, L2 → info, L3 →
    success).
- **BaseModal ×1**: the inline edit-user modal becomes
  `<BaseModal v-model:open="…" title="Editar @username">`.
- **BaseButton ×3+**: refresh button (ghost sm), edit-modal cancel
  (ghost) and save (primary) buttons.
- Drop scoped CSS absorbed by the atoms: `.bigstat-row` row
  layout stays (grid wrapper), but `.bigstat` cell rules go;
  `.data-table` / `.tbl` rules go; `.status-dot` / `.status-dot.*`
  go; `.kyc-badge` / `.kyc-l0..l3` go; `.modal-overlay` / `.modal`
  / `.modal-h` / `.modal-body` / `.modal-actions` go; `.btn-ghost`
  / `.btn-primary` go.

### Non-goals

- **No migration of `UserDrawer.vue`, `WalletDrawer.vue`,
  `ConfirmUserModal.vue`.** These three are the v1 drawer trio
  named in DESIGN-v2 §5 #5 as Phase 7.6 (drawer unification).
  AdminUsersView keeps importing them in their v1 shape — only the
  AdminUsersView's own scoped chrome migrates.
- **No `BulkActionBar` primitive, no row-selection.** DESIGN-v2
  §5 #3 mentions "bulk-action footer" as a deliverable here, but
  7.1.c §1 non-goaled the primitive and 7.1.c §10 punted it to a
  follow-up (`SelectableTable` extension). Ratifying the punt:
  bulk-action lands in a 7.4.1 follow-up after the primitive
  ships from a dedicated 7.1.x.
- **No `<row-actions>` consumption.** Same rationale — the kebab-
  menu pattern that surfaces ban/freeze/delete shortcuts per row
  is most useful once UserDrawer no longer opens by default on
  row click (i.e. once 7.6 lands). Until then, row-click → open
  drawer is the canonical action path.
- **No `TabGroup` primitive.** The 5 filter tabs stay as inline
  `<button class="tab">` markup. Tabs surface in 4 places across
  the codebase today (chart ranges, table toolbars, drawer tabs,
  settings nav); the 5th consumer triggers extraction. Marked as
  a tracked follow-up in §10.
- **No `Pager` primitive.** The paging row stays inline. Same
  rationale — Pager lifts when a second consumer (AdminWalletsView
  in Phase 7.5, probably) surfaces.
- **No `BaseInput` / `BaseSelect`.** The toolbar's `<input>` and
  two `<select>` controls stay native + scoped CSS. Phase 7.6 or
  7.x will surface enough form-field consumers to drive the API.
- **No backend changes.** All admin API endpoints
  (`listUsers`/`banUser`/`updateUser`/etc.) stay as-is.
- **No new dependency.**

## 4. Column metadata (BaseTable)

```ts
const userColumns: Column<AdminUser>[] = [
  { key: 'user',         label: 'Usuario' },
  { key: 'status',       label: 'Estado' },
  { key: 'kyc',          label: 'KYC' },
  { key: 'country',      label: 'País' },
  { key: 'wallets',      label: 'Wallets',      num: true },
  { key: 'balance',      label: 'Saldo total',  num: true },
  { key: 'last_active',  label: 'Última actividad' },
  { key: 'created_at',   label: 'Registro' },
]
```

Slot map:

| Column key | Slot | Content |
|---|---|---|
| `user` | `cell-user` | Avatar (`<div class="avatar">` with hue/initials) + display-name + username/email subtitle |
| `status` | `cell-status` | `<BaseBadge :tone>` (active/banned/deleted) |
| `kyc` | `cell-kyc` | `<BaseBadge variant="outline" :tone>` (L0..L3) |
| `country` | `cell-country` | Flag emoji + ISO code (kept inline) |
| `wallets` | (default) | `userWalletCount(u.user_id)` plain number |
| `balance` | `cell-balance` | `<span class="mono">{{ userTotalLabel(u.user_id) }}</span>` |
| `last_active` | `cell-last_active` | `<span class="muted">{{ relTime(u.last_active) }}</span>` |
| `created_at` | `cell-created_at` | `<span class="muted">{{ fmtDate(u.created_at) }}</span>` |

Row-click handler: `@row-click="(p) => openDrawer(p.row)"` —
preserves the existing v1 behavior of clicking a row to open
`UserDrawer`. The drawer itself stays v1 until Phase 7.6.

`rowClass`: returns `'row-muted'` when `u.deleted_at` is set
(preserves the v1 visual cue that the deleted user is dimmed).

## 5. Status / KYC tone mapping

### Status → BaseBadge tone (soft variant, dot enabled)

| v1 class | v1 label | BaseBadge tone |
|---|---|---|
| `status-dot.active` | "Activo" | `success` |
| `status-dot.banned` | "Baneado" | `danger` |
| `status-dot.deleted` | "Eliminado" | `neutral` |

### KYC level → BaseBadge tone (outline variant, no dot)

| Level | Tone | Rationale |
|---|---|---|
| `L0` | `warning` | Unverified — needs attention |
| `L1` | `warning` | In progress |
| `L2` | `info` | Verified, mid-tier |
| `L3` | `success` | Fully verified |

The mapping is heuristic; if reviewer requests a different tone
distribution, flip the table without amending the spec. The
relevant invariant is "L0/L1 read as 'attention needed', L2/L3
read as 'OK'".

## 6. Behavior

- **Row click**: opens `UserDrawer` (existing v1). BaseTable
  emits `@row-click` with `{ row, index, event }`; the handler
  unwraps `row` and calls `openDrawer(row)`.
- **Edit modal**: opens via the existing `openEdit(u)` flow from
  the drawer or row-context-menu. `<BaseModal v-model:open>` is
  controlled by `editTarget` being non-null; closing the modal
  (`update:open=false` or scrim/ESC) clears `editTarget`.
- **Filter tabs + search + paging**: behavior unchanged. The
  same `filterTab` / `searchQuery` / `currentPage` refs drive
  the same `filteredUsers` / `pagedUsers` computeds. Only the
  rendering changes.
- **Loading state**: existing `loading` ref still gates the
  inline `.loading-row` indicator above the table. BaseTable
  doesn't have a built-in loading state; if the migration
  reveals a UX need for one, ship it as a 7.1.c amendment.
- **Empty state**: BaseTable's `<template #empty>` renders the
  "No se encontraron usuarios con los filtros actuales." message
  when `pagedUsers.length === 0`.

## 9. Acknowledged deviation from DESIGN-v2 §5 #3

DESIGN-v2 §5 #3 names three deliverables for Phase 7.4:

1. **`BaseTable` adoption** — ✓ in scope.
2. **Row-actions** — deferred to Phase 7.6 (alongside drawer
   unification, when the action-path layer is rationalised).
3. **Bulk-action footer** — deferred to a future 7.1.x sub-phase
   that ships `BulkActionBar` + `SelectableTable` extension to
   BaseTable. 7.1.c §10 already tracks this as a follow-up.
4. **Redesigned filter chips** — deferred; the current filter
   tabs stay inline. The redesign lifts when `TabGroup` /
   `FilterChips` primitive surfaces from a multi-consumer trigger.

Ratifying these deferrals in 7.4's spec rather than amending
DESIGN-v2 because:

- Each deferred item is its own contract surface (Pager,
  BulkActionBar, TabGroup, row-actions UX layer) that warrants
  its own 7.1.x sub-phase with proper SDD discipline.
- Bundling them into 7.4's screen migration would mix screen-
  migration work with primitive extraction — exactly the kind of
  mixing DESIGN-v2 §6 ("one screen or one base component per PR,
  never two") prohibits.
- The visual-refresh deliverable (BaseTable + KPI cards + status
  badges) is the high-leverage win; the rest can land
  incrementally without blocking the screen-migration cadence.

## 10. Out-of-scope follow-ups

### Phase 7.x candidates (named in §9)

- **`Pager` primitive**: lift when AdminWalletsView (Phase 7.5)
  surfaces the same paging shape. Two consumers is enough.
- **`TabGroup` primitive**: lift on the 5th consumer (tabs
  appear in 4 places today).
- **`BulkActionBar` + `SelectableTable` extension**: dedicated
  sub-phase (7.1.x) after BaseModal/Drawer land. Likely between
  7.6 and 7.7.
- **Row-actions UX layer**: Phase 7.6 — the kebab-menu per row
  pattern + the action-confirmation flow (ban / freeze / delete)
  surface together with the drawer migration.

### Composable extraction

- **`useFilteredPagedList`**: the `filterTab` / `searchQuery` /
  `currentPage` / `filteredUsers` / `pagedUsers` / `totalPages`
  block on AdminUsersView is the second occurrence of a filter +
  paging pattern (AdminAuditView has a similar one in v1). Lift
  to a composable when the 3rd consumer surfaces.

### Token follow-ups

- The `cursor: pointer` row hint (rows clickable to open drawer)
  is an inline style on the v1 template. BaseTable does not
  expose a "rows are clickable" hint; if the consumer-driven
  cursor cue feels universal across screens, surface a
  `BaseTable :interactive` prop in a follow-up amendment.

### Legacy cleanup

- `.bigstat` / `.data-table` / `.btn` / `.modal-overlay` rules in
  `src/assets/main.css` stay reachable for the other v1 admin
  views (`AdminWalletsView`, `AdminAuditView`, etc.). Last
  consumer removes them per DESIGN-v2 §6.
