## 1. Goal & non-goals

### Goal

Close the three gaps left open by 7.10.d:

1. **Visibility** — a treasury distribution sits in `pending_approval` until a second admin approves it, but `/admin/sends` only iterates blocks + mempool, so it's invisible. The user who creates a send can't see what they just did.
2. **Action** — there is no UI to approve or cancel a pending distribution. Approval also requires a *different* admin with `APPROVE_TREASURY_DISTRIBUTION` granted; the spec must document the testing path.
3. **Visual fidelity** — `NewSendFlow.vue` (7.10.d) skipped the slide-to-confirm animation and the rich receipt layout from the v2 design reference (`propuesta_rediseno_blockchain_2/flows.jsx:586-730`, mirrored in `SendConfirmFlow.vue`). The existing Vue component already implements the design faithfully but is decoupled from the new flow.

### In scope

- Backend: extend `GET /admin/sends` to merge `pending_approval` distributions as virtual rows.
- Frontend: Aprobar / Cancelar buttons on `SendDetailModal` when `row.status === 'pending' && row.kind === 'treasury'`, gated by the appropriate permission.
- Frontend: extract `SlideToConfirm.vue` atom from `SendConfirmFlow.vue` and reuse it in `NewSendFlow.vue` step 1; port the rich receipt layout (check circle, big mono amount, key/value card) to step 2.
- Docs: short snippet in this spec on how to create a second admin for testing the dual-sign path.

### Non-goals

- Cancelling executed distributions (irreversible by design).
- Modifying the dual-sign rule (`approver_id != initiator_id` is sacred).
- Re-writing `SendConfirmFlow.vue` itself — we extract the slide widget but leave the existing component in place for its current consumers (P2P, mobile sheets).
- Realtime updates (still gated on Phase G.6).

## 3. Sub-phase 7.10.e.a — Pending sends visibility (backend)

### Endpoint change

`GET /admin/sends` (Phase 7.10.a) must merge pending distributions into the response so the user sees what they initiated.

### Row shape mapping

A pending distribution carries no `Transaction`, so we synthesize a SendRow per recipient:

```python
for d in treasury_distributions.list(status="pending_approval"):
    for recipient_user_id in d.recipient_user_ids:
        yield SendRow(
            tx_id=f"dist:{d.op_id}:{recipient_user_id}",
            ref_short=d.op_id[:14],
            kind="treasury",
            status="pending",
            confirmations=None,
            from=<resolved from d.source_wallet_id>,
            to=<resolved from recipient_user_id + d.currency>,
            asset={"code": d.currency, "symbol": d.currency},
            amount=str(d.amount_per_wallet),
            amount_usd=<converted>,
            fee="0",
            fee_usd="0",
            note=d.memo,
            confirmed_at=None,
            created_at=d.initiated_at,
            # New field — only present on virtual rows so the frontend
            # can render Aprobar/Cancelar:
            distribution={
                "op_id": d.op_id,
                "initiated_by": d.initiated_by,
                "is_pending_approval": True,
            },
        )
```

### `SendRow` schema additions (frontend mirror)

Add an optional `distribution?: { op_id, initiated_by, is_pending_approval }` field. Real (Transaction-backed) rows leave it unset.

### Aggregates impact

- `count_24h`: virtual rows count toward total if `initiated_at` is in window.
- `count_24h_by_kind.treasury`: bump by virtual rows.
- `pending_onchain`: stays as-is (counts only mempool-backed pending), but rename the KPI to **"Pendientes"** to cover both flavours; the footer changes from "Confirmaciones < 3" to "Aprobación o confirmación".

### Filter behaviour

- `status=pending` includes virtual rows.
- `kind=treasury` includes virtual rows.
- `q=` matches against `op_id` short form.

### Tests

- `tests/test_admin_sends.py`: add cases for virtual rows surfacing in the list, KPI bump, and status/kind filters.

## 4. Sub-phase 7.10.e.b — Approve / Cancel actions (frontend)

### Detail modal

Extend `SendDetailModal.vue` footer when `row.distribution?.is_pending_approval`:

| Actor                              | Buttons              |
| ---------------------------------- | -------------------- |
| `row.distribution.initiated_by === me.user_id` | Cancelar (red ghost) |
| `me has APPROVE_TREASURY_DISTRIBUTION` AND `initiated_by !== me` | Aprobar (primary) |
| Otherwise                          | None (read-only)     |

### Wire

- `Cancelar` → `cancelTreasuryDistribution(op_id)` → toast + close + reload table.
- `Aprobar` → `approveTreasuryDistribution(op_id)` → toast (mentions transactions now in mempool) + close + reload.

### Permission gate sourcing

There is **no** GET endpoint for the caller's effective permissions today. Three options:

1. Add `GET /admin/users/<self>/permissions` in a separate phase.
2. Optimistically render the button and translate 403 into a clear inline error.
3. Use `auth.hasRole('ADMIN')` as a proxy and live with the imperfect gate.

**Decision:** **(2) optimistic + translated 403.** Mirrors the same UX pattern already used in `AdminSendsView` for `VIEW_TRANSFERS`. Adding a permissions read endpoint is its own phase; not blocking 7.10.e.

### Tests

- Vitest: `SendDetailModal` renders Aprobar when `row.distribution` present and current user is not the initiator; renders Cancelar otherwise.

## 5. Sub-phase 7.10.e.c — Slide-to-confirm port + receipt polish

### Goal

Replace the current step-1 confirm card in `NewSendFlow.vue` with the v2 slide-to-confirm widget, and step-2 receipt with the rich layout (success circle, big mono amount).

### Plan

1. **Extract `src/components/atoms/SlideToConfirm.vue`** from the existing `SendConfirmFlow.vue` slide block (lines 113–135 + the JS handlers).
   - Props: `disabled?: boolean`.
   - Emits: `confirm` once `drag > 0.92` and release happens.
   - Keep the visual exactly as in `SendConfirmFlow.vue` (verified against `propuesta_rediseno_blockchain_2/flows.jsx:658-685`).
2. **NewSendFlow step 1** uses `SlideToConfirm` AND keeps the "Confirmar y enviar" button (button calls the same submit handler). The slide is the primary affordance; the button is the fallback (matches the React reference: "o tocá el botón para confirmar").
3. **NewSendFlow step 2** mirrors React `flows.jsx:692-712`: 72px success circle with check icon, negative mono amount, "Recibió tu envío" copy, kv card with Operación / Hora / Saldo restante.
4. **No refactor of `SendConfirmFlow.vue`** — it stays as-is; it's used by `P2PBuyFlow` and the mobile sheets.

### Out of scope

- Sound effects, haptics, complex spring animations — the CSS transition in the React reference is the spec.
- Accessibility for the slide: ship a sensible keyboard fallback (Enter on focused thumb triggers `confirm`), but full WCAG audit is its own phase.

## 6. Documentation — testing the dual-sign path locally

```powershell
# 1. Register a second admin (bootstrap or promote via existing endpoints).
# 2. As ADMIN A (the initiator), self-grant the permissions you need:
$body = '{"action":"grant","permission":"APPROVE_TREASURY_DISTRIBUTION"}'
Invoke-RestMethod -Method Post `
  -Uri "http://localhost:3000/api/v1/admin/users/<ADMIN_B_USER_ID>/permissions" `
  -Headers @{ Authorization = "Bearer <ADMIN_A_JWT>" } `
  -ContentType "application/json" -Body $body

# 3. Log in as ADMIN B, open /admin/sends, click the pending row,
#    press Aprobar. The row flips to Pendiente (on-chain) until the
#    next mine_block, then Completado.
```

A short paragraph in `docs/api-reference.md` should cross-reference this path.

## 7. Acceptance criteria

1. A pending treasury distribution surfaces as a row in `/admin/sends` with `kind=treasury`, `status=pending`, and the new `distribution.is_pending_approval = true` flag.
2. `count_24h` and `count_24h_by_kind.treasury` include pending distributions.
3. Clicking the row opens `SendDetailModal` with Aprobar/Cancelar buttons rendered per the actor table in §4.
4. Cancelling reflects in the table immediately after the toast.
5. Approving as ADMIN_B (different from initiator) generates the N mempool transactions and (after the next mine) the row appears as Completado, with `distribution` dropped.
6. `NewSendFlow.vue` step 1 shows the slide-to-confirm widget matching `SendConfirmFlow.vue` 1:1; step 2 mirrors `flows.jsx:692-712`.
7. The `SlideToConfirm` atom is reusable (lands in `src/components/atoms/`).
8. CI green, no regressions, vue-tsc strict passes.
