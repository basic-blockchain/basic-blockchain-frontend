## 1. Goal & non-goals

### Goal

Bring `/admin/sends` to the v2 reference design: real KPIs, currency column with asset badge, classified `kind` (Interno / On-chain / Tesorería), detail modal per row, and a fully wired "Nuevo envío" flow (form → confirm with slide-to-confirm → receipt).

### In scope

- `src/views/AdminSendsView.vue` — refactor (data source, KPIs, columns).
- `src/api/admin.ts` — wrapper for `/admin/sends`.
- New `src/components/organisms/SendDetailModal.vue`.
- New `src/components/organisms/NewSendFlow.vue` (treasury-only path, wraps `SendConfirmFlow.vue`).
- New `src/components/atoms/AssetBadge.vue` — currency pill extracted from `AdminWalletsView` / `AdminCurrenciesView`.
- Reuse existing `UserChip.vue` for `De` / `Para` columns (avatar + name).
- Backend: derive `kind` server-side and expose 24h/7d window filters.

### Non-goals

- User-facing "my sends" view (separate phase).
- "Send on behalf of <user>" (case B) — requires new `TRANSFER_ON_BEHALF` permission and approval flow; deferred to its own phase.
- "Send from admin's own wallet" (case A) — already covered by `/wallets/me`; redundant here.
- Realtime WebSocket updates (depends on Phase G.6).
- CSV export, custom date ranges, advanced filters.
- Refactor of the existing `SendConfirmFlow.vue` (reused as-is for steps 2–3).
- Block/tx explorer deep-link — "Ver en explorer" button stubbed if route missing.

### "Nuevo envío" semantics (decided)

The `[+ Nuevo envío]` button in `/admin/sends` opens a **treasury-only** send flow (case C). Source wallet is always a treasury wallet (Hot/Cold) selected at step 1. This integrates with the dual-sign envelope from [7.8.0-treasury-dual-sign](../../../basic-blockchain-simulator/docs/specs/7.8.0-treasury-dual-sign.md): admins with `CREATE_TREASURY_DISTRIBUTION` initiate; admins with `APPROVE_TREASURY_DISTRIBUTION` approve. The flow is **not** a personal wallet send.

## 3. Data contract

### 3.1 Endpoint

`GET /admin/sends?window=24h|7d|all&kind=internal|onchain|treasury&status=completed|pending|failed&q=<text>&page=<n>`

Auth: admin role.

### 3.2 Response row (`SendRow`)

```ts
interface SendRow {
  tx_id: string;            // canonical id (used as ref short form)
  ref_short: string;        // e.g. "ref_a82bc4" or "0x4f1a…d77e"
  kind: 'internal' | 'onchain' | 'treasury';
  status: 'completed' | 'pending' | 'failed';
  confirmations: number | null;   // null for non-onchain
  from: { user_id: string|null; username: string|null; address: string };
  to:   { user_id: string|null; username: string|null; address: string };
  asset: { code: 'cUSD'|'USDT'|'BTC'|'USDC'|string; symbol: string };
  amount: number;
  amount_usd: number;
  fee: number;
  fee_usd: number;
  note: string | null;
  confirmed_at: string | null;    // ISO-8601, null if pending
  created_at: string;             // ISO-8601
}
```

### 3.3 Aggregates response

```ts
interface SendsAggregates {
  count_24h: number;
  count_24h_by_kind: { internal: number; onchain: number; treasury: number };
  volume_usd_24h: number;
  pending_onchain: number;        // kind=onchain AND confirmations<3
  fees_collected_usd_7d: number;
}
```

Returned alongside list (`{ rows, aggregates, pagination }`).

### 3.4 Classification rules (server-side)

| Condition                                                                | `kind`     |
| ------------------------------------------------------------------------ | ---------- |
| `from.user_id != null` AND `to.user_id != null` AND neither is treasury  | `internal` |
| Either `from` or `to` wallet is in `treasury_wallet_ids` set (Hot/Cold)  | `treasury` |
| Either side has `user_id == null` (external address) or `tx.onchain==true` | `onchain`  |

`treasury_wallet_ids` is loaded from config / DB (existing Phase 7.8 treasury concept).

### 4.3 New-send flow (`NewSendFlow` — 3 steps)

**Step 1 — Form** (new):
- **Wallet origen** (select de wallets de tesorería disponibles, filtradas por currency)
- Destinatario (autocomplete por @username — usuario interno; o address externa si la wallet de tesorería soporta on-chain)
- Activo (derivado de la wallet origen — no editable manualmente)
- Monto (input num, muestra USD live, valida saldo disponible de la wallet origen)
- Nota (textarea opcional, queda en audit_log)
- Footer: `Cancelar` · `Continuar`

**Step 2 — Confirm** (reuse `SendConfirmFlow` step 0):
- Avatar + nombre destinatario
- Monto, Valor USD, Comisión (`Gratis` para internos), Tiempo estimado, Nota
- Slide-to-confirm O botón `Confirmar y enviar`

**Step 3 — Receipt** (reuse `SendConfirmFlow` step 1):
- ✓ "¡Enviado!"
- Monto negativo + "Recibió tu envío"
- Operación (TXN-XXXXX), Hora, Tu saldo restante
- `Cerrar` · `Compartir recibo`

On step-3 close → emit `complete` → parent toasts + reloads table.

## 6. Acceptance criteria

1. KPIs match: counts in last 24h, volume in USD, pending-onchain (<3 conf), fees collected 7d.
2. `Tipo` column shows three distinct values (`Interno` / `On-chain` / `Tesorería`) on representative seed data.
3. `Activo` column renders currency badge for each row.
4. Click on any row opens `SendDetailModal` with all fields populated.
5. `[+ Nuevo envío]` opens form → confirm → receipt; on success the new row appears in the table after close.
6. Filters (`Todos / Internos / On-chain / Pendientes / Fallidos`) apply both client-side (current rows) and update the URL `?kind=` for shareable links.
7. Backend unit tests cover all three `kind` classification branches.
8. No `any` types introduced; strict TS passes.
9. Responsive: at <640px, `Hash/Ref` and `Cuando` columns hidden; KPI grid collapses to 1 column.
10. **Permissions**:
    - Page `/admin/sends` requires `VIEW_TRANSFERS` (ADMIN must have it granted explicitly per [V009](../../../basic-blockchain-simulator/migrations/versions/V009__roles_permissions.sql); OPERATOR has it baseline; VIEWER does not).
    - `[+ Nuevo envío]` button visible only if user has `CREATE_TREASURY_DISTRIBUTION`.
    - The confirm step in the flow triggers the dual-sign envelope (7.8.0). Approval by a second admin with `APPROVE_TREASURY_DISTRIBUTION` is required before the row appears as `completed`; until then it shows as `pending`.
