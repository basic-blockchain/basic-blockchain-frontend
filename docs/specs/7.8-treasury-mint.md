## 1. Goal & non-goals

### Goal

Seventh screen-phase of v2. Migrates `src/views/AdminTreasuryView.vue`
to atoms AND wires the **Stepper `error` state** into
`TreasuryApprovalFlow.vue` for the rejection path — the
proactively-shipped status from 7.1.d gets its first real consumer.

### In scope

- **AdminTreasuryView migration** (230 LOC):
  - 4 bigstat KPIs → `<BaseCard variant="bigstat">`
  - Treasury wallets table → `<BaseTable>` with cell-<key> slots
  - Currency badge → `<BaseBadge variant="outline" tone="warning">`
  - "Distribuir" / "Actualizar" / "Crear" buttons → `<BaseButton>`
- **TreasuryApprovalFlow rejection path**: add a `rejectedStep`
  state. When the 2nd approver rejects, the relevant step in the
  Stepper flips to `status: 'error'`. The flow exposes a "Rechazar"
  affordance on the 2nd-approval step that consumers can wire to
  the rejection action.
- **Currency badge tone**: warning (operational attention) per
  the convention established in 7.5 (TREASURY → warning).

### Non-goals

- **No backend changes.** The dual-sign endpoints proposed in
  DESIGN-v2 §7 are a separate simulator-side phase. The rejection
  path here is a local UX state (flow-internal), not a real
  approval-status fetch. When the simulator ships
  `/api/v1/admin/treasury/distribute/{op_id}/approve` and the
  `REJECTED` return path, frontend wiring lands in 7.8.1.
- **No new Stepper amendment.** The `error` state shipped in 7.1.d
  is consumed as-is. If the visual treatment of "error" feels
  insufficient during review (e.g., reviewer wants an explicit
  rejection icon vs. the current `!` glyph), amend 7.1.d separately.
- **No mint flow extension.** DESIGN-v2 §7.2 names
  "Mint approval (extension of `/admin/mint`)" as a backend
  proposal. The current AdminView mint form is single-signer
  (Phase 6e); the dual-sign wrap happens when simulator ships
  the `MINT_DUAL_SIGN_THRESHOLD` knob. Out of scope here.
- **No new dependency.**

## 3. Migration map

### AdminTreasuryView

Same shape as Phase 7.3 / 7.4 / 7.5:
- bigstat KPIs → `<BaseCard variant="bigstat">`
- panel + table → `<BaseCard variant="default" padding="none">`
  + `<BaseTable>`
- currency-badge → `<BaseBadge variant="outline" tone="warning">`
- buttons → `<BaseButton>`

### TreasuryApprovalFlow rejection path

Add to script setup:
```ts
const rejectedStep = ref<number | null>(null)

const stepsWithStatus = computed(() =>
  signupSteps.map((s, i) => ({
    ...s,
    status: i === rejectedStep.value ? 'error' as const : undefined,
  })),
)

function reject() {
  rejectedStep.value = step.value
}
```

Template:
- Pass `:steps="stepsWithStatus"` (instead of plain `signupSteps`).
- Add a "Rechazar" affordance on the 2nd-approval step (only
  visible when `step === 2` and `rejectedStep === null`).
- On reject: the Stepper flips that step to `error` (red `!`
  glyph), the flow's footer shows a "Operación rechazada" banner,
  and the user can close the flow without proceeding.
