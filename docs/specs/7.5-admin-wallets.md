## 1. Goal & non-goals

### Goal

Fourth screen-phase of v2. Migrates `src/views/AdminWalletsView.vue`
(396 LOC) to consume the Phase 7.1 atom set with a twist: it is
the **first real consumer of BaseTable's `row-actions` slot**
(designed in 7.1.c §5 but unused until now). The v1 already
renders the freeze/unfreeze toggle as a trailing column with an
empty header — a 1:1 fit for the slot.

### In scope

- Migrate `src/views/AdminWalletsView.vue` only.
- **BaseCard (bigstat) ×4**: Total / Activas / Congeladas / Saldo
  bajo gestión. Same pattern as AdminUsers (Phase 7.4) and
  AdminView (7.3) so the three admin pages share KPI typography.
- **BaseTable ×1**: 8 data columns + `row-actions` slot for the
  freeze/unfreeze toggle. The slot renders inside an
  empty-header trailing column, which is the exact contract 7.1.c
  §5 specified.
- **BaseBadge ×4 axes**:
  - **Status** (active / frozen): tone `success` for active, tone
    `info` for frozen. Soft variant + dot.
  - **Type** (USER / TREASURY / FEE): tone `neutral` /
    `warning` / `info`. Soft variant, no dot (it's a type tag,
    not a status state).
- **BaseButton ×6+**:
  - Refresh + Exportar stub + "Congelar selección" stub → ghost sm.
  - Per-row freeze/unfreeze toggle → secondary sm (text carries
    the semantic).
- Drop scoped CSS for atom-absorbed patterns: `.bigstat`,
  `.data-table` + descendants, `.status-dot.*`, `.type-badge.*`,
  `.btn-sm`, `.btn-success`, `.btn-info`. Keep view-specific
  CSS: `.asset-pill` (AssetPill follow-up), `.filter-input` /
  `.filter-select`, `.usd-cell`, `.vl-native`, `.username`,
  `.display-name`.

### Non-goals

- **No migration of `WalletDrawer.vue`** — Phase 7.6 unifies it
  with UserDrawer onto BaseDrawer.
- **No `Pager` primitive extraction**. The §10 follow-up in 7.4
  named Phase 7.5 as the Pager-extraction trigger because
  "AdminWalletsView likely has the same paging shape" —
  re-verified during 7.5 spec drafting: **AdminWalletsView does
  NOT have paging in v1**, it renders all filtered rows. Pager
  extraction waits for a third consumer.
- **No `AssetPill` primitive extraction**. The `.asset-pill`
  recipe (36px min-width tinted box with currency code) is one
  of the DESIGN-v2 §4 "orden a definir" atoms. Defer until at
  least one other screen consumes it.
- **No `BaseInput` / `BaseSelect`**. The filter bar's
  `<input>` and `<select>` controls stay native + scoped CSS,
  same as Phase 7.4.
- **No real implementation of the two stub buttons** ("Exportar",
  "Congelar selección"). They were stubs in v1 and stay stubs
  here — the migration only swaps their chrome to BaseButton.
- **No backend changes.**
- **No new dependency.**

## 3. Anatomy

### Top of view (KPIs)

Identical pattern to AdminUsers (7.4) and AdminView (7.3) bigstat
rows. Four `<BaseCard variant="bigstat">` cells.

The fourth KPI ("Saldo bajo gestión") is the densest: it shows
USD aggregate when priced, falls back to a native multi-currency
string when no FX rate exists, and finally renders `$0` when no
balances. The fallback text uses `.vl-native` scoped class to
reduce font size — keep that scoped class as a cell-internal
override.

### Filter bar (kept inline)

`<input>` + `<select>` + count badge stays as scoped chrome.
Behavior unchanged.

### Data table (BaseTable + row-actions)

```
┌─[ BaseTable ]──────────────────────────────────────────────────────┐
│ Activo │ Wallet ID │ Usuario │ Balance │ USD │ Tipo │ Pub key │ Estado │ (actions) │
│ ──────────────────────────────────────────────────────────────────│
│ USDT   │ wal_…AB   │ Alice   │ 100.0   │ $100│ USER │ 0xfa…91 │ ●Activa│ [Congelar]│
│ ──────────────────────────────────────────────────────────────────│
│ …                                                                  │
└────────────────────────────────────────────────────────────────────┘
```

Column 9 is **not** a `columns[]` entry — it is the BaseTable
`row-actions` slot, rendered automatically when the slot is
provided.

### Row actions slot scope

```vue
<template #row-actions="{ row }">
  <BaseButton variant="secondary" size="sm" @click="toggleFreeze(row)">
    {{ row.frozen ? 'Descongelar' : 'Congelar' }}
  </BaseButton>
</template>
```

The row's own click handler (open drawer) is suppressed inside
this slot because the BaseButton consumes the click event before
it bubbles to the `<tr>`. This is the exact mechanism v1 used
with `@click.stop` on the last `<td>`. BaseTable does not need
a special "actions don't open the drawer" prop — event
propagation handles it.

## 4. Column metadata (BaseTable)

```ts
interface WalletColumn {
  key: string
  label: string
  num?: boolean
}
const walletColumns: WalletColumn[] = [
  { key: 'asset',    label: 'Activo' },
  { key: 'wallet_id',label: 'Wallet ID' },
  { key: 'user',     label: 'Usuario' },
  { key: 'balance',  label: 'Balance' },
  { key: 'usd',      label: 'USD',     num: true },
  { key: 'type',     label: 'Tipo' },
  { key: 'pub_key',  label: 'Clave pública' },
  { key: 'status',   label: 'Estado' },
]
```

Slot map:

| Column key | Slot | Content |
|---|---|---|
| `asset` | `cell-asset` | `<span class="asset-pill">{{ row.currency }}</span>` (kept inline pending AssetPill primitive) |
| `wallet_id` | `cell-wallet_id` | `<HashChip :hash="row.wallet_id" :length="16" label="wallet id" />` (existing atom) |
| `user` | `cell-user` | display-name + @username inline |
| `balance` | `cell-balance` | `<AmountDisplay :amount :precision="8" :unit="row.currency" />` (existing atom) |
| `usd` | `cell-usd` | `${{ formatUsd(row.balance_usd) }}` or em-dash placeholder when null |
| `type` | `cell-type` | `<BaseBadge :tone>` per type (USER/TREASURY/FEE) |
| `pub_key` | `cell-pub_key` | `<HashChip :hash="row.public_key" :length="14" label="public key" />` |
| `status` | `cell-status` | `<BaseBadge :tone>` (success for active, info for frozen) |

Plus the `row-actions` slot (§3).

### Row click + click.stop

Cells that contain interactive elements (HashChip copy buttons,
the freeze toggle button) need their click events stopped from
bubbling to the `<tr>` so the drawer doesn't open. The v1
template uses `@click.stop` on those `<td>`s; BaseTable's slot
content can wrap in `<span @click.stop>` or rely on the
interactive elements themselves to call `event.stopPropagation()`.

The HashChip atom already stops propagation internally for its
copy button click. The BaseButton in `row-actions` does NOT need
explicit stop — its click handler emits before the bubble reaches
`<tr>`, and the row-click event on BaseTable fires only when the
`<tr>` itself receives the event. Native button semantics handle
this.

For the explicit `@click.stop` on cells (wallet_id and pub_key
columns), wrap the slot content in `<div @click.stop>` to
preserve the v1 behavior.

## 6. Behavior

- **Row click → openDrawer(w)** preserved via BaseTable's
  `@row-click` event.
- **Freeze toggle** preserved via the `row-actions` slot
  consuming `toggleFreeze(row)`. Native button-click semantics
  prevent bubbling.
- **Filter / search** behavior unchanged. The `filtered`
  computed feeds BaseTable's `rows` prop directly.
- **`rowClass`**: returns `'row-muted'` when `w.frozen` is true
  (preserves the v1 muted-row visual for frozen wallets).

  later.

### New in 7.5

- **`AssetPill` primitive**: the `.asset-pill` recipe on this
  view is the first known consumer. The DESIGN-v2 §4 "orden a
  definir" lists this atom. Lift when at least one other screen
  (likely WalletView or ExchangeView in Phase 7.x) surfaces the
  same shape.
- **Row-action propagation policy in BaseTable**: this PR
  empirically validates that native button click semantics
  suppress the row-click bubble. If a consumer needs a non-
  button row-action (e.g., an `<a>` link or a custom div), the
  pattern documented here (wrap in `@click.stop`) holds. No
  amendment needed.
