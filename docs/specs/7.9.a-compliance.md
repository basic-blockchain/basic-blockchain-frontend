## 1. Goal & non-goals

### Goal

First long-tail sub-phase of v2. Migrate the compliance cluster views
(Audit, Compliance queue, KYC admin review) to consume Phase 7.1 atoms
without changing data sources or behavior.

### In scope

- `src/views/AdminAuditView.vue`
- `src/views/AdminComplianceView.vue`
- `src/views/AdminKycView.vue`

Shared migration rules (all three views):
- Bigstat KPIs -> `<BaseCard variant="bigstat">`
- Inline tables/lists -> `<BaseTable>` (where a row list exists)
- Status chips -> `<BaseBadge>` (tone mapping defined per view)
- Inline buttons -> `<BaseButton>`
- Drop scoped CSS for `.bigstat*`, `.tbl*`, `.btn*`, `.panel*`, and any
  per-view chip styles that become BaseBadge.

### Non-goals

- No backend/API changes.
- No router changes (routes stay the same).
- No new dependency.
- No new composable extraction beyond the optional TabGroup check
  defined in Section 4.
- No migration of non-compliance views (7.9.b/7.9.c/7.9.d are separate
  specs).

## 4. Props / structure changes

### 4.1 AdminAuditView -> BaseTable columns

```ts
const columns = [
  { key: 'created_at', label: 'Cuando' },
  { key: 'actor', label: 'Actor' },
  { key: 'event', label: 'Evento' },
  { key: 'origin', label: 'IP / Origen' },
]
```

Slot map:
- `cell-created_at`: `mono` timestamp
- `cell-actor`: actor id + category badge
- `cell-event`: action + detail
- `cell-origin`: placeholder dash (unchanged)

Category badge -> `BaseBadge` with tone mapping:
- `security` -> `danger`
- `treasury` -> `warning`
- `kyc` -> `success`
- `wallet` -> `info`
- `user` -> `info`
- `permission` -> `info`
- `p2p` -> `warning`
- `role` -> `warning`
- `config` -> `neutral`
- fallback -> `neutral`

### 4.2 AdminComplianceView -> BaseTable columns

Queue becomes a BaseTable with slot-heavy cells to preserve layout.

```ts
const columns = [
  { key: 'priority', label: '' },
  { key: 'user', label: 'Usuario' },
  { key: 'kind', label: 'Tipo' },
  { key: 'detail', label: 'Detalle' },
  { key: 'risk', label: 'Riesgo' },
  { key: 'age', label: 'Antiguedad' },
  { key: 'actions', label: '' },
]
```

Slot map:
- `cell-priority`: dot indicator (keep as inline div)
- `cell-user`: avatar initial + name
- `cell-kind`: `BaseBadge` (tone from kind mapping)
- `cell-detail`: detail text
- `cell-risk`: `BaseBadge` (tone from risk mapping)
- `cell-age`: time since
- `row-actions`: `BaseButton` (Revisar)

Kind tone mapping (existing logic):
- AML / Sanciones -> `danger`
- Disputa -> `warning`
- KYC -> `info`

Risk tone mapping:
- Critico -> `danger`
- Alto -> `warning`
- Medio -> `info`
- Bajo -> `success`

### 4.3 AdminKycView -> BaseCard + BaseBadge

- KPI row -> BaseCard bigstat x3.
- Review card container -> `<BaseCard>` (default) per review.
- Level pills -> `BaseBadge` (soft, dot=false):
  - current level -> `neutral`
  - pending level -> `info`
- Document status -> `BaseBadge` (soft):
  - verified -> `success`
  - rejected -> `danger`
  - pending_review -> `warning`
  - missing -> `neutral`
- Action buttons -> BaseButton:
  - Approbar -> `primary`
  - Rechazar -> `danger`
  - Promover -> `primary`

### 4.4 TabGroup trigger check

## 5. Behavior

- All data fetches and stores stay unchanged.
- Search + filter logic in AdminAuditView remains identical.

- KYC approval/rejection/promote flows remain identical.
- Row click behavior only changes where BaseTable rows are used (no

## 6. Accessibility

- Ensure tab strip uses `role="tablist"` and each tab `role="tab"`
  with `aria-selected` bound to the active key.
- All icon-only buttons include visible labels or `aria-label`.
- BaseTable header labels remain text, not icon-only.
- Badge text remains visible (no color-only state).
