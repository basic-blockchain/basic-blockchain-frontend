## 1. Goal & non-goals

### Goal

Reshape `src/components/atoms/BaseBadge.vue` from a domain-coupled
`status` enum API to a visual-primitive `tone × variant × dot` API, so
every Phase 7.2+ screen can consume badges by tone (the visual
contract) instead of by status (a domain concern). Anchored in
DESIGN-v2 §4 row 2.

### In scope

- Rewrite `BaseBadge.vue` props to `{ tone, variant, dot?, ... }`.
  Drop the existing `status` prop entirely (no compat shim).
- Migrate the existing in-tree consumers of the OLD `BaseBadge` API
  in the same commit as the API change, doing only the mechanical
  prop rename (`:status="…"` → `:tone="…"` per the mapping table in
  §4). This is NOT a screen migration; chrome stays v1.
- Retokenize the badge styles to use `--radius-pill` (Phase 7.0)
  instead of the inlined `border-radius: 999px`.
- Ship unit + snapshot tests covering `tone × variant × dot`.

### Non-goals

- **`StatusBadge.vue` stays.** It is a divergent older component
  (RGBA hex colors, uppercase labels, used by `HealthView`). It
  migrates inside Phase 7.7 (ChainView / MempoolView / NodesView /
  HealthView) — not here. Two badge components coexist on develop
  until 7.7, which is acceptable: they target different visual
  registers (StatusBadge is the v1 health chrome).
- **No migration of inline `.bdg` usages** in `src/views/*` or
  prototype-style spans (e.g., compliance "KYC L2" pills).
  Those become `<BaseBadge>` only when their screen migrates in
  Phase 7.4 / 7.7 / 7.9.
- **No new tones beyond the six in §4.** Custom inline colors that
  appear in the prototype (e.g., `#5b21b6` for staff, OFAC red) are
  captured by tone `accent` (purple/brand) and tone `danger`; no
  per-domain tones.
- Removing the legacy `.bdg*` rules from `src/assets/main.css` is
  deferred to the last screen migration, per DESIGN-v2 §6.

## 3. Anatomy

```
┌─[ span.base-bdg ]─────────────────────────────────┐
│  ┌──────┐  ┌────────────────────────────────────┐ │
│  │ dot  │  │ default slot (label)               │ │
│  │ (•)  │  │                                    │ │
│  └──────┘  └────────────────────────────────────┘ │
└───────────────────────────────────────────────────┘
    ↑ gap: 5px (matches prototype `.bdg { gap: 5px }`)

Dot is a `::before` pseudo-element so consumers cannot accidentally
push it out of position with extra slot content. Hidden via
`display: none` when `dot=false` or `variant='outline'` default.
```

Token map (host element):
- `border-radius`: `var(--radius-pill)` (99px).
- `padding`: `2px 7px 2px 6px` (soft) · `2px 7px` (outline, no dot).
- `gap`: `5px` (raw — no Phase 7.0 token at this dimension).
- `font`: `500 11px/1.5 var(--font-sans)`.
- `white-space`: `nowrap`.
- Background, color, dot color: variant + tone (see §4).

## 4. Variants & states

### Tones

Six tones, each backed by an existing token pair (`*-soft` + `*-text`
or `*-soft` + `*`). No raw hex values introduced.

| Tone | Soft bg | Soft text | Dot color | Source v1 status mapping |
|---|---|---|---|---|
| `success` | `--success-soft` | `--success` | `--success` | `active`, `ok` |
| `warning` | `--warning-soft` | `--warning` | `--warning` | `pending_kyc`, `degraded` |
| `danger`  | `--danger-soft`  | `--danger`  | `--danger`  | `banned`, `error` |
| `info`    | `--info-soft`    | `--info`    | `--info`    | `frozen` |
| `neutral` | `--muted-soft`   | `--muted`   | `--muted`   | `deleted` |
| `accent`  | `--accent-soft`  | `--accent-text` | `--accent` | `staff`, plus prototype "kind" chips ("KYC L2", "AML alert"). Replaces the inlined `#ede9fe`/`#5b21b6` of the prototype's `.bdg-staff` — a Phase 7.0 token-only choice. |

### Variants

| Variant | Description | Tokens |
|---|---|---|
| `soft` (default) | Filled-soft pill with dot. The default shape inherited from `.bdg-active` et al. | bg + text from tone, dot visible |
| `outline` | Surface-2 bg + 1px border. No dot by default. Mirrors prototype `.bdg-kyc`. | `background: var(--surface-2); color: var(--text-2); border: 1px solid var(--border)`. The `tone` prop on outline tweaks the **text** color only (e.g., `<BaseBadge tone="danger" variant="outline">` = surface-2 chip with danger-colored text). |

### `dot` prop

| Variant | Default `dot` | Override |
|---|---|---|
| `soft` | `true` | `dot=false` hides the `::before` dot, keeps padding |
| `outline` | `false` | `dot=true` enables the dot; coloured by tone |

### States

| State | Trigger | Visual |
|---|---|---|
| `idle` | default | as per tone × variant |
| (no hover / focus / active / disabled) | — | BaseBadge is **non-interactive** by default. It is a `<span>`, not a `<button>`. Consumers that need a clickable badge wrap in `<BaseButton iconOnly>` or `<button>` themselves. |

## 5. Props / Slots / Events

### Props (TypeScript)

```ts
type Tone = 'success' | 'warning' | 'danger' | 'info' | 'neutral' | 'accent'
type Variant = 'soft' | 'outline'

interface BaseBadgeProps {
  tone?: Tone           // default: 'neutral'
  variant?: Variant     // default: 'soft'
  dot?: boolean         // default: variant === 'soft'
}
```

### Slots

| Slot | Purpose | Notes |
|---|---|---|
| (default) | Label text | Required-by-convention. If omitted, the badge renders an empty pill — useful only for stand-alone dot indicators (rare). |

### Events

None. BaseBadge is a non-interactive span. Fallthrough attrs apply
on the root `<span>` (e.g., `role`, `aria-label`, `data-testid`).

## 6. Behavior

- Pure render. No state. No watchers. No emits.
- The `::before` dot is hidden via CSS, not via `v-if`, so toggling
  `dot` between renders does not cause layout reflow.
- `outline` variant overrides the host background and adds a border;
  the tone-class still applies (controls text color).

---

## 7. Accessibility

- Root is `<span>`. No `role="status"` by default — the existing
  BaseBadge.vue sets it implicitly but for static labels (e.g.,
  "Active" in a table cell) `role="status"` causes screen readers
  to announce on initial render in some flows, which is noisy.
  Consumers that need live announcements pass `role="status"`
  through fallthrough attrs.
- The dot is a `::before` pseudo-element with no semantic content;
  the tone information must also be conveyed by the slot text or a
  consumer-provided `aria-label`. The component does NOT inject an
  `aria-label` derived from `tone` (the tone is a visual register,
  not a label — "danger" is rarely the right reader-announced word
  for "Banned").
- Contrast: every (tone, variant) cell uses existing v1 tokens that
  already meet WCAG AA, verified in Phase 5e against the same
  `*-soft` × `*-text` pairs.

## 10. Out-of-scope follow-ups

### Resolved before impl (closed during spec review)

- **Padding kept as raw `2px 7px 2px 6px`.** The asymmetry is
  intentional — it compensates for the 6px dot pseudo-element on
  the left so the optical center matches the right edge. Tokenizing
  to `var(--space-xs) var(--space-sm)` (4px 6px) would re-center the
  pill and break the prototype's optical alignment. Revisit only if
  BaseTable's row rhythm forces a different vertical metric.
- **No default `role="status"`.** Confirmed per §7. Consumers that
  need live-region announcements pass `role` through fallthrough.

### 7.1 follow-up components

- `CountChip` — small numeric pill used in tabstrip counts
  (`<span className="count">23</span>` in compliance-screens.jsx).
  Distinct primitive (smaller font, different padding, no tone).
  Probably 7.1.x or absorbed into `BaseTable` toolbar.
- `EnvPill` — the "Producción" / "Staging" topbar pill
  (`.env-pill` in `styles.css`). Same visual recipe as
  `outline + dot`, but anchored to env state. Likely absorbed by
  BaseBadge `<BaseBadge variant="outline" dot tone="warning">Prod</BaseBadge>`
  when the topbar migrates in Phase 7.2.

### Token follow-ups

- The raw `5px` gap and `2px 7px 2px 6px` padding have no Phase 7.0
  tokens. Tokenize only if a second small-pill primitive needs the
  same values — premature otherwise.

### Legacy cleanup

- Inline `.bdg` consumers in `src/views/*` and prototype-style
  inline-styled `.bdg` spans migrate per their screen-phase
  (7.4 / 7.7 / 7.9). The legacy `.bdg*` CSS in
  `src/assets/main.css` is deleted in the PR that migrates the
  LAST inline consumer, per DESIGN-v2 §6.
- `StatusBadge.vue` consolidation into BaseBadge is a Phase 7.7
  decision (it ships with HealthView migration).
