## 1. Goal & non-goals

### Goal

First screen-phase of v2. Migrates the three auth views
(`LoginView.vue`, `RegisterView.vue`, `ActivateView.vue`) to consume
the Phase 7.1 base atoms (BaseButton + Stepper + BaseBadge) and
extracts the duplicated `auth-wrap` chrome into a reusable
`<AuthLayout>` molecule.

DESIGN-v2 §5 #1 names "Login / Register (Phase 7.2)" as the first
visible v2 surface — single endpoint, high credibility gain.

### In scope

- **Migrate three view files in one PR**:
  - `src/views/LoginView.vue`
  - `src/views/RegisterView.vue`
  - `src/views/ActivateView.vue`
- **Extract `src/components/molecules/AuthLayout.vue`**: the
  two-column auth shell (left form panel + dark right "platform
  pitch" panel + brand + foot). Slots:
  - default → left-panel form content
  - `right-panel` → dark-side content (optional, defaults to the
    standard quote + role-list block already on `LoginView`)
- **Consume `<BaseButton>`** for the submit buttons (Login,
  Register, Activate) and any "back" / "cancel" buttons. Drop the
  scoped `.btn-primary` / `.auth-btn-outline` CSS rules that
  duplicate the BaseButton recipe.
- **Consume `<Stepper>`** to replace the hand-rolled
  `<div class="progress-bar">` 3-step indicator on Register and
  Activate. Use the new `Step[]` shape from 7.1.d.
- **Consume `<BaseBadge>`** (outline variant, neutral tone) to
  replace the `<div class="progress-label">` "Paso N de M" text in
  Register and Activate. The Stepper renders the visual cursor;
  the badge is the textual label.
- **No legacy scoped CSS left behind**: the auth-related `.btn-*`
  / `.progress-*` / `.fld` rules that AuthLayout absorbs are
  deleted in this PR; the v1 versions in
  `src/assets/main.css` stay until the last auth-CSS consumer
  migrates.

### Non-goals

- **No BaseCard consumption in Phase 7.2.** DESIGN-v2 §5 #1 names
  Card in the "exercises every base component" framing, but the
  Login visual contract has no light-surface card: the role-list
  pattern lives on a dark gradient panel that does not fit
  BaseCard's `var(--surface)` recipe. BaseCard is first consumed
  in Phase 7.3 (AdminView dashboard) where the bigstat variant
  is first-class. This deviation from §5 #1 is acknowledged and
  ratified by this spec — no amendment to DESIGN-v2 needed.
- **No `AuthPlatformPanel` extraction.** The dark right-panel
  with role-list lives only on `LoginView` today. Extracting it
  for a single consumer is premature. If Register and Activate
  later adopt the same panel, lift the extraction in a follow-up.
- **No new input primitive (`BaseInput`).** The `.fld` form rows
  carry domain-specific markup (labels, optional hints, country
  selects). Extracting them is a Phase 7.x follow-up after enough
  consumers surface to drive the contract. The `.fld` scoped CSS
  moves into `AuthLayout` so the three views consume it through
  the layout, not re-define it per view.
- **No router or auth-store changes.** Pinia / Vue Router /
  `src/api/auth.ts` stay as-is. This is a visual / structural
  refactor only.
- **No new dependency.**
- **No screen-level Playwright smoke.** A manual-screenshot pair
  in the PR description satisfies DESIGN-v2 §6 ("a manual
  screenshot pair in the PR description otherwise"). Playwright
  introduction is itself a separate decision (Phase 7.1.a §6
  flagged it as a deferred choice).

```vue
<AuthLayout>
  <form class="auth-form" @submit.prevent="submit">
    <Stepper :steps="signupSteps" :current="currentStepIndex" />
```

array defined inline per view (no premature extraction). The two
views differ only in `currentStepIndex`.

## 6. Behavior

- **No store / API changes.** The `auth` store, `useAuth.login`,
  `register`, `activate` keep their signatures. The submit
  handlers and the toast-on-error logic stay verbatim.
- **Loading state** flows through to `<BaseButton :loading>`.
  BaseButton replaces both the spinner glyph (`pi-spin
  pi-spinner`) and the conditional label text — the spinner sits
  in the leading slot; the label slot stays `Continuar`. The v1
  pattern `{{ loading ? 'Ingresando…' : 'Continuar' }}` is
  intentionally dropped: BaseButton's loading state already
  signals the busy condition; the label text staying constant
  reads cleaner.
- **Disabled state** is implicit when `loading=true` (BaseButton
  contract); no `:disabled="loading"` is needed in addition.
- **Stepper status mapping**:
  - Register's `currentStepIndex = 0` (Datos de acceso).
  - Activate's `currentStepIndex = 1` (Contraseña).
  - After Activate completes, `/login` is reached — no third-step
    rendering of the Stepper is needed because Login is a
    different view.
- **Responsive breakpoint** stays at 760px (matches the v1
  scoped media query). Single source of truth in `AuthLayout`.

  - renders the default slot inside `.auth-left`

## 10. End of "base atoms" phase, start of "screens" phase

Phase 7.2 is the inflection point between sub-phase 7.1 (base
component construction) and sub-phases 7.3..7.9 (screen
migrations). Notable shifts in shape from here onward:

- **PRs grow in LOC, shrink in API surface.** A screen migration
  rewrites a view but does not introduce new contracts.
- **Manual validation matters more.** Pre/post screenshots become
  the canonical review evidence; unit tests cover the new
  components, not the view itself.
- **The "atomic API swap" exception no longer applies.** Screen
  migrations swap one view-shape for another; there is no
  cross-file API contract at risk.
- **Spec → impl PR pairing stays.** Every screen-phase still
  lands a spec before code, per the SDD discipline established in
  Phase 7.1.
