## 1. Goal & non-goals

### Goal

Add a 3-step `Nuevo usuario` wizard to `AdminUsersView` so admins can create users (especially **other admins**) directly from the UI. Closes the dependency on `curl`/Postman to test the dual-sign flow shipped in 7.10.e.

### In scope (this phase)

- Wizard with 3 steps:
  1. **Identidad** — display name, username (auto-suggested from display name), email (optional), country (optional).
  2. **Rol y KYC** — role radio (VIEWER / OPERATOR / ADMIN), KYC level select.
  3. **¡Listo!** — surfaces `activation_code` + instructions for the new user to activate.
- `+ Nuevo usuario` button in `AdminUsersView` header.
- After completion, refresh the user list and toast.

### Non-goals (deferred to 7.11.b)

- Permissions override step (4th step in React reference).
- Invitation step with email send.
- KYC document upload.
- Phone field (backend doesn't store it today).
- Setting an initial password — only the user can, via `/auth/activate`.

## 2. Reference

- React source: `docs/propuesta_rediseno_blockchain_2/flows-onboarding.jsx:425-540` (steps 0–1 layout).
- Backend endpoints:
  - `POST /auth/register` ([api/auth_routes.py:52](../../../basic-blockchain-simulator/api/auth_routes.py)) accepts `{username, display_name, email?, country?}` and returns `{user_id, username, activation_code}`.
  - `POST /admin/users/<id>/roles` ([api/admin_routes.py](../../../basic-blockchain-simulator/api/admin_routes.py)) accepts `{action: 'grant', role}`. New users land as VIEWER by default; if the admin selected OPERATOR or ADMIN, the wizard calls this immediately after register.
- Frontend wrappers already exist: `register`, `grantRole`. No new client code needed.

## 5. Acceptance criteria

1. `+ Nuevo usuario` button visible on `AdminUsersView` (no permission gate for this phase — `AdminUsersView` is already gated by `VIEW_USERS` at the route level).
2. Submitting step 1 with role=ADMIN calls `register` then `grantRole(<new_id>, 'ADMIN')`.
3. Step 2 shows the activation code in a copyable monospace pill.
4. On close (after success), the user list reloads and a success toast fires.
5. Cancel from step 0 or 1 closes without side-effects.
6. Username auto-derives from `first_name.last_name` lowercased (matches the React reference) but can be edited.
7. CI green, no regressions.

## 6. KYC field — note

The backend stores `kyc_level` per user but the `/auth/register` payload doesn't accept it (today). Two options for this phase:

1. **Drop the KYC field from the wizard** — set it via a follow-up admin call (`PATCH /auth/me/profile`? Not as admin). Cleanest backend story.
2. **Show the KYC select read-only as "L0 (inicial)"** and document that the admin can elevate via a separate phase.

**Decision: (2).** Show the select disabled with "L0 · sin verificar". Honest with the user that we set the initial level for them; the elevation path is a separate phase (likely 7.12 KYC admin).
