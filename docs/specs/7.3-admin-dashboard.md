## 1. Goal & non-goals

### Goal

Second screen-phase of v2. Migrates `src/views/AdminView.vue` (the
admin dashboard built in Phase 6e) to consume the full Phase 7.1
atom set: BaseCard (default + bigstat variants), BaseBadge, BaseTable,
BaseButton, and Stepper. Adds the "stepper-style refresh indicator"
called out in DESIGN-v2 §5 #2.

This phase is the **first real BaseCard consumer**, paying back the
deferral made in Phase 7.2 §1.

### In scope

- Migrate `src/views/AdminView.vue` (581 LOC) only. NOT
  `DashboardView.vue` (user-side portfolio — its own future phase).
- **BaseCard (`bigstat` variant) ×4**: replace the four KPI tiles
  in the `.bigstat-row` (Usuarios / Activos / Wallets /
  Transacciones·7d) with `<BaseCard variant="bigstat">`. Each
  consumes the three slots: `header` (label + optional trend
  badge), default (the big numeric value), `footer` (the
  descriptive `.ds` subtitle).
- **BaseCard (default variant) ×5**: replace `<section class="panel">`
  wrappers on Volume chart, Critical events, Top movements,
  Blockchain KPIs, Mint form, and Balances. Each uses the `header`
  slot for the panel title (and right-side controls where present)
  and the default slot for content.
- **BaseBadge ×3** (outline + tone): the three trend pills (users
  / active / tx) become `<BaseBadge variant="outline" :tone>` with
  tone derived from `deltaTone()` (up→success, down→danger,
  flat→neutral).
- **BaseTable ×1**: the Top movements table becomes `<BaseTable>`
  with typed `Column<TopMovement>[]` and a `cell-from-to` slot for
  the composite from→to cell.
- **BaseButton ×3**: the two page-action buttons ("Ver usuarios",
  "Ver auditoría") become `<BaseButton variant="ghost" size="sm">`;
  the mint submit becomes `<BaseButton variant="primary"
  :loading="minting" type="submit">`.
- **Stepper refresh indicator** (DESIGN-v2 §5 #2): a horizontal
  Stepper rendered above the bigstat row during the initial parallel
  fetches. Each step represents one fetch (stats / metrics / volume
  / top-movements / critical-events). Status mapping per step:
  - `pending` while the fetch has not started (instantaneous in
    practice because all five fire from `Promise.all`)
  - `current` while the fetch is in-flight
  - `done` after the fetch resolves successfully
  - `error` after the fetch rejects (consumer maps caught errors
    to per-step status)
  - When all five are `done` or `error`, the Stepper fades out
    over `var(--duration-base)`. A manual "Refresh" button
    re-triggers it.
- **Drop scoped CSS** for the patterns absorbed by atoms:
  `.bigstat-row`, `.bigstat`, `.bs-h`, `.lb`, `.vl`, `.ds`,
  `.trend-pill`, `.tone-up/-down/-flat`, `.panel`, `.panel-h`,
  `.panel-loading`, `.panel-empty`, `.btn`, `.btn-primary`,
  `.btn-ghost`, `.tbl` and its descendants (`thead`, `.num`,
  `.movement-pair` cell wrapper kept, etc.).

### Non-goals

- **No migration of `DashboardView.vue`.** User-portfolio dashboard
  is a separate phase (likely Phase 7.x within the "long tail" set).
- **No `BaseInput` extraction.** The `.field` / `.field-input` /
  `.field-label` rows on the mint form are the 4th occurrence of a
  custom input shape (after Login/Register/Activate). DESIGN-v2 §4
  does not list BaseInput; defer until Phase 7.4 / 7.5 surface a
  consistent shape.
- **No `TabGroup` primitive** for the `.range-tabs` segmented
  control on the Volume chart. The recipe lives in 4 places
  (admin charts, table toolbars) — extracting needs at least one
  more consumer to validate the contract. Inline scoped CSS stays.
- **No `BaseChart` wrapper.** `<VChart>` from `vue-echarts` stays;
  the volume chart panel is wrapped in `<BaseCard>` but the chart
  itself is not abstracted.
- **No backend changes** to dashboard endpoints (`/dashboard/volume`,
  `/dashboard/top-movements`, `/admin/audit`). The Stepper consumes
  the existing fetch promises; it does not introduce new requests.
- **No new dependency.**

### Stepper refresh indicator state model

```ts
type FetchKey = 'stats' | 'metrics' | 'volume' | 'top-movements' | 'critical-events'
type FetchStatus = 'pending' | 'current' | 'done' | 'error'

const fetchStatus = reactive<Record<FetchKey, FetchStatus>>({
  stats: 'pending',
  metrics: 'pending',
  volume: 'pending',
  'top-movements': 'pending',
  'critical-events': 'pending',
})

const refreshSteps = computed(() => ([
  { key: 'stats',           label: 'Stats',     status: fetchStatus.stats           },
  { key: 'metrics',         label: 'Metrics',   status: fetchStatus.metrics         },
  { key: 'volume',          label: 'Volumen',   status: fetchStatus.volume          },
  { key: 'top-movements',   label: 'Top mov',   status: fetchStatus['top-movements'] },
  { key: 'critical-events', label: 'Eventos',   status: fetchStatus['critical-events'] },
]))

const refreshCurrent = computed(() =>
  refreshSteps.value.findIndex(s => s.status === 'current'),
)

const refreshComplete = computed(() =>
  refreshSteps.value.every(s => s.status === 'done' || s.status === 'error'),
)
```

The Stepper renders while `!refreshComplete`. The fade-out is a
`v-if` + Vue `<Transition name="fade">` wrapper.

## 6. Behavior

- **`onMounted` orchestration**: replace `Promise.all([loadX, loadY,
  …])` with a per-fetch wrapper that sets `fetchStatus[key] =
  'current'` before the fetch and flips to `'done'` or `'error'`
  after. `Promise.allSettled` (not `Promise.all`) so a single
  rejection doesn't short-circuit the rest. Per-fetch toast errors
  stay (existing behavior).
- **Manual refresh**: a "Refresh" button (`<BaseButton
  variant="ghost" size="sm">`) above the bigstat-row resets all
  five `fetchStatus[*]` to `'pending'`, then re-runs the same
  orchestrator. The Stepper fade-in/out follows
  `refreshComplete` automatically.
- **Stepper visibility**: while `!refreshComplete`. Once all five
  are terminal (`done` or `error`), the Stepper exits via Vue
  `<Transition>` (fade over `var(--duration-base)`).
- **Initial-load race**: all five fetches fire concurrently from
  `onMounted`, so `pending` is observable only between the
  reactive ref initialization and the first microtask. In practice
  the user never sees `pending` — they see four `current` and one
  starting to resolve. Stepper still reads correctly because the
  visual contract from 7.1.d treats `pending` as "not started"
  (no dot fill) and `current` as the cursor.

## 7. Test plan

### Unit tests

- **`useDashboardFetchSteps.spec.ts`** (new composable): factor the
  per-fetch wrapper + `fetchStatus` state into
  `src/composables/useDashboardFetchSteps.ts` so it's unit-testable
  without mounting AdminView. Test cases:
  - initial status is `'pending'` for all 5 keys
  - calling `run(key, fetcher)` flips status to `'current'`,
    awaits, then `'done'` on resolve
  - `run(key, fetcher)` flips to `'error'` on reject without
    re-throwing past the wrapper boundary (so `Promise.allSettled`
    works)
  - `refreshComplete` is `true` only when every key is `done` or
    `error`
  - `reset()` flips all keys back to `'pending'`

No new view-level tests on AdminView itself (consistent with the v1
testing strategy that doesn't unit-test screen surfaces).

### Snapshot tests

None new. AdminView is a screen surface and v1 has no snapshot
test for it; introducing one in 7.3 would balloon the matrix
without a compelling regression-detection win.

### Manual validation (PR description checklist)

- `npm run dev` → `/admin`:
  - Initial load: Stepper visible above bigstat row, animates
    through the 5 steps as each fetch resolves.
  - Once all five resolve: Stepper fades out, the dashboard
    renders the bigstat KPIs + chart + tables.
  - Click "Refresh" → Stepper re-appears, walks through again.
  - Force one fetch to fail (intercept the API in DevTools): the
    Stepper step turns `error` (red `!`), other steps continue
    normally, toast fires; the refresh-complete state holds when
    all five are terminal.
- Light + dark theme: bigstat cards read at WCAG AA; trend
  badges in both themes match the prototype.
- Pre/post screenshot pair (light + dark, viewport 1440px = 4
  screenshots).
