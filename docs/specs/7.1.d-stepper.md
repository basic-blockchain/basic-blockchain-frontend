## 1. Goal & non-goals

### Goal

Reshape `src/components/atoms/Stepper.vue` from the v1 string-array
API to a v2 object-array API per DESIGN-v2 §4 row 7. The component
remains a horizontal multi-step indicator used by approval / KYC /
treasury flows; the new shape adds stable `key`s and per-step
`status` overrides (including a new `error` state) so consumers can
reflect mid-flow failures without re-deriving from `current`.

DESIGN-v2 §5 #7 names this primitive as a prerequisite for the
Treasury & Mint approval flow in Phase 7.8.

### In scope

- Rewrite `src/components/atoms/Stepper.vue` props to
  `{ steps: Step[], current: number }` where
  `Step = { key: string; label: string; status?: StepStatus }`.
- New `StepStatus` enum: `'pending' | 'current' | 'done' | 'error'`.
  Default derivation from `current`:
  `i < current → done`, `i === current → current`, `i > current → pending`.
  Per-step `status` overrides the derivation when set.
- Migrate the three in-tree consumers in the same commit as the
  rewrite (mechanical `['a','b','c']` → `[{key:'a',label:'a'},...]`):
  `src/components/flows/P2PBuyFlow.vue`,
  `src/components/flows/WithdrawFlow.vue`,
  `src/components/flows/TreasuryApprovalFlow.vue`.
- Retokenize hardcoded `0.15s` transitions to
  `var(--duration-fast)` + `var(--ease-out)`.
- Ship unit + snapshot tests covering status derivation, cursor
  position, status override, and the error variant.

### Non-goals

- **No vertical orientation.** Every existing consumer renders
  horizontal; vertical lands as 7.1.x only when a flow asks for it.
- **No "skipped" status.** The prototype does not model it; revisit
  if Phase 7.8 approval flows require optional / branching steps.
- **No clickable steps / step navigation.** Stepper stays a passive
  indicator. Consumers that want clickable history wrap each label
  in their own anchor.
- **No screen-migration of the three flows.** This PR only updates
  their Stepper invocation; the rest of each flow stays on v1
  chrome until Phase 7.6 (drawers) and 7.8 (approval flows).

## 4. Status × visual mapping

| Status | Circle bg | Circle color | Circle border | Label color | Label weight | Connector entering this step |
|---|---|---|---|---|---|---|
| `done` | `var(--success)` | `#faf9f6` (light theme) / `var(--bg)` token | transparent | `var(--text)` | 500 | `var(--success)` |
| `current` | `var(--text)` | `var(--bg)` | `2px solid var(--text)` | `var(--text)` | 600 | `var(--success)` if any prior step is `done`, else `var(--border)` |
| `pending` | `var(--surface-2)` | `var(--text-3)` | `1px solid var(--border)` | `var(--text-3)` | 500 | `var(--border)` |
| `error` | `var(--danger)` | `#faf9f6` / `var(--bg)` token | transparent | `var(--danger)` | 600 | `var(--danger)` if entering from `done` or `error`, else `var(--border)` |

Circle contents:
- `done` → `✓` glyph.
- `error` → `!` glyph.
- `current` / `pending` → 1-based step index (`i + 1`).

`#faf9f6` is the prototype's literal `--bg` light-theme value. To
stay token-correct, the spec mandates `var(--bg)` (which already
resolves to `#faf9f6` on light theme and a dark surface on dark
theme). Removing the inlined hex from the existing v1 component is
one of the cleanups the rewrite picks up.

## 5. Props / Slots / Events

### Props (TypeScript)

```ts
type StepStatus = 'pending' | 'current' | 'done' | 'error'

interface Step {
  key: string
  label: string
  status?: StepStatus
}

interface StepperProps {
  steps: Step[]                // required, at least 1 entry
  current: number              // required, 0-based cursor index
}
```

### Slots

None. The label is rendered from `Step.label` directly. If a
consumer needs richer content (icons, two-line labels) it surfaces
in 7.1.x via a `label-<key>` slot; not in 7.1.d.

### Events

None. The component is a passive indicator. Consumers drive
`current` and per-step `status` externally.

## 6. Behavior

- **Status resolution order**: explicit `Step.status` wins; if
  absent, derived from `current`:
  `i < current → done`, `i === current → current`, `i > current →
  pending`.
- **Multiple `current` steps**: the component does NOT enforce
  uniqueness. If two steps have explicit `status: 'current'`, both
  render as current; same for connectors. Consumers control this.
- **Error after current**: a step with explicit `status: 'error'`
  past the `current` index is allowed (e.g., "the second approval
  failed before the first one completed" is a valid model in
  treasury flows). The connector entering that error step picks the
  rule in §4.
- **Empty steps array**: a runtime guard in `<script setup>` logs a
  dev warning. Rendering an empty stepper is allowed but useless.

## 7. Accessibility

- Wrapper is a `<ol>` (ordered list) with
  `role="list"` retained-by-default. Each step is an `<li>` with
  `aria-current="step"` on the current step.
- The circle's glyph (`✓` / `!` / index) is decorative — wrapped in
  `aria-hidden="true"`. The step's accessible label is `Step.label`
  rendered as visible text inside the `<li>`.
- Connectors are decorative `<div>`s with `aria-hidden="true"`.
- No focusable elements inside the stepper (no clickability per §1).
- Status is conveyed by text on screen readers via the label only.
  If a status distinction matters semantically (e.g., "error" must
  be announced), the consumer should include that meaning in the
  label string itself — e.g., `label: 'Firmar (falló)'` when the
  consumer sets `status: 'error'`. Documenting in the consumer-side
  call-site, not enforced by Stepper.

## 8. Test plan

### Unit tests (`Stepper.spec.ts`)

- renders one `<li>` per step
- assigns the derived class `is-done` to steps with `i < current`
- assigns the derived class `is-current` to the step at index
  `current`
- assigns the derived class `is-pending` to steps with `i > current`
- per-step `status` override wins over derivation (e.g., a step
  with `status: 'error'` at `i < current` renders `is-error`, not
  `is-done`)
- renders `✓` glyph for `done`, `!` for `error`, and `i + 1` for
  `current` / `pending`
- sets `aria-current="step"` on the resolved-current step only
- emits no events (sanity-check the empty `defineEmits`)
- renders connectors between every adjacent pair (N-1 connectors
  for N steps)
- connector entering a `done` step receives `is-done` connector
  class; entering a `pending` step receives `is-pending`; entering
  an `error` step from a `done` step receives `is-error`
- logs a dev warning when `steps` is empty

### Snapshot test (`Stepper.snap.spec.ts`)

Six snapshots covering structural axes:

1. 4 steps · `current = 0` (no done yet).
2. 4 steps · `current = 2` (two done + one current + one pending).
3. 4 steps · `current = 3` (last step active).
4. 5 steps · `current = 4` (all done — the "complete" terminal state).
5. 4 steps · `current = 2` with the current step explicitly overridden
   to `error` (the future treasury-failure case).
6. 5 steps · `current = 1` with step 2 explicitly overridden to
   `done` (skip-ahead override case).

### Manual validation (PR description checklist)

- Light + dark theme: each status renders at expected contrast.
- The three migrated flows (P2PBuy, Withdraw, TreasuryApproval)
  open and render identically pre/post-migration in their idle
  paths.
- TreasuryApprovalFlow: when an approver hypothetically rejects
  (consumer test code can simulate by overriding `status: 'error'`),
  the visual diff matches the §4 mapping.

## 10. Out-of-scope follow-ups

### Resolved before impl (closed during spec review)

- **`current` stays required** rather than derived from per-step
  status. Aligns with DESIGN-v2 §4 literal and with the three
  existing consumers (all drive `current` as a single
  `ref<number>`). Derivation-only would force the three consumers
  into a more verbose `steps.map` per step transition.

### 7.1 follow-up components

- **Vertical Stepper.** No prototype reference today; lift to 7.1.x
  only when a flow asks. The DOM model would change (the connector
  becomes a vertical line on the left rail), so it is closer to a
  sibling component than a variant prop on the same SFC.
- **`label-<key>` slot.** If 7.8 treasury flows need approver
  avatars inline with the step label, ship the slot as an additive
  amendment (no breaking change on existing consumers).

### Token follow-ups

- `margin-bottom: 18px` on the wrapper is a raw value. Tokenize
  only when a second multi-step primitive (vertical Stepper,
  Wizard) needs the same vertical rhythm.
- The 22px circle dimension matches the inline 22px in
  `propuesta_refactorizacion/flows.jsx`. No token yet; revisit if a
  larger / smaller Stepper variant lands.

### Legacy cleanup

- None. Stepper has no parallel v1 component (unlike StatusBadge ↔
  BaseBadge).
