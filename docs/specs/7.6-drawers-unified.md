## 1. Goal & non-goals

### Goal

Fifth screen-phase of v2. Migrates the four overlay components
in `src/components/drawers/` and `src/components/modals/` onto the
Phase 7.1.e BaseDrawer / BaseModal engine, eliminating the
duplicated Teleport + scrim + focus-trap + body-scroll-lock + ESC
machinery each one currently hand-rolls.

### In scope

- Migrate **all four** overlay files:
  - `src/components/drawers/UserDrawer.vue` (808 LOC) ‚Üí
    `<BaseDrawer>` shell + internal tab/section slots kept as
    consumer markup.
  - `src/components/drawers/WalletDrawer.vue` (403 LOC) ‚Üí
    `<BaseDrawer>` shell.
  - `src/components/drawers/ProfileDrawer.vue` (649 LOC) ‚Üí
    `<BaseDrawer>` shell.
  - `src/components/modals/ConfirmUserModal.vue` (188 LOC) ‚Üí
    `<BaseModal>` shell.
- **Single PR** for all four because they share the same v1 CSS
  recipes (`.scrim`, `.drawer`, `.drawer-head`, `.drawer-body`,
  `.modal-scrim`, `.modal`) and migrating them piecemeal would
  leak v1 chrome onto develop between PRs.
- **Drop legacy CSS** from `src/assets/main.css` for any
  `.scrim` / `.drawer` / `.modal-scrim` / `.modal` rules that
  the four migrated components were the last consumers of.
  Verified via grep before deletion.

### Non-goals

- **No breadcrumb-style nav between drawer levels.** DESIGN-v2
  ¬ß5 #5 names this as a deliverable here:
  > "group into a single detail-panel pattern with breadcrumb-
  > style navigation between related entities (user ‚Üí wallet ‚Üí
  > tx)"
  Implementing it requires a new state machine (current entity
  + history stack), a new UI element (breadcrumb header inside
  the drawer), and changes to consumer call-sites (AdminUsers
  needs to wire "open wallet inside UserDrawer instead of
  side-by-side"). That is genuinely new behavior, not chrome
  swap. Defer to a **Phase 7.6.1 follow-up** so 7.6's PR stays
  reviewable as a mechanical migration. Same precedent as
  7.4 ¬ß9 (row-actions, bulk-action, filter chips ‚Üí follow-ups).
- **No internal-content refactor** in the four drawers. Each
  drawer's tabs, sections, action buttons, and field layouts
  stay verbatim. Only the outer shell + the scrim/Teleport
  machinery move.
- **No `BaseInput` / `BaseSelect` / `BaseTextarea`.** Each
  drawer carries a few inputs; they stay native + scoped CSS,
  consistent with prior phases.
- **No backend / API changes.** All `freezeWallet` / `banUser` /
  `softDeleteUser` / etc. callsites keep their existing
  signatures.
- **No new dependency.** Focus trap + scroll lock come from
  7.1.e's hand-rolled implementation.

## 3. Migration shape (per file)

Each of the four files follows the same pattern:

### Before (v1 sketch)

```vue
<template>
  <Teleport to="body">
    <div :class="['scrim', { open }]" @click="emit('close')" />
    <aside :class="['drawer', { open }]" role="dialog" aria-label="‚Ä¶">
      <header class="drawer-head">‚Ä¶</header>
      <div class="drawer-body">‚Ä¶content‚Ä¶</div>
    </aside>
  </Teleport>
</template>

<style scoped>
.scrim   { /* lifted into BaseDrawer */ }
.drawer  { /* lifted into BaseDrawer */ }
.drawer-head { /* kept as consumer chrome */ }
</style>
```

### After (7.6)

```vue
<template>
  <BaseDrawer :open="open" @update:open="emit('close')">
    <template #header>
      <div class="drawer-head">‚Ä¶</div>
    </template>
    <div class="drawer-body">‚Ä¶content‚Ä¶</div>
  </BaseDrawer>
</template>

<style scoped>
/* .scrim / .drawer rules deleted ‚Äî BaseDrawer owns the shell. */
/* .drawer-head / .drawer-body / .section-h / etc. kept ‚Äî those
 * are consumer-chrome rules that style the drawer's internal
 * sections. */
</style>
```

The same shape applies to `ConfirmUserModal.vue` using
`<BaseModal>` instead of `<BaseDrawer>`.

### Prop wiring per component

| Component | BaseDrawer/Modal props | Consumer event |
|---|---|---|
| `UserDrawer` | `<BaseDrawer :open :title="user?.fullName" :width="720">` | `@update:open="(v) => v || emit('close')"` |
| `WalletDrawer` | `<BaseDrawer :open :title="wallet wallet_id summary" :width="640">` | same |
| `ProfileDrawer` | `<BaseDrawer :open :title="'Perfil'" :width="540">` | same |
| `ConfirmUserModal` | `<BaseModal :open :title="config.title" :width="480">` | same; cancel/confirm buttons go in `<template #footer>` |

The `title` prop is the simplest a11y surface ‚Äî when the consumer
doesn't pass one, the v1 hand-rolled header inside the slot still
provides accessible context via its own `<h2>` markup.

### Width values

- 720px for UserDrawer (matches the v1 width + matches the
  BaseDrawer default).
- 640px for WalletDrawer (slightly narrower ‚Äî wallet detail is
  less dense than user detail).
- 540px for ProfileDrawer (form-shaped, narrower works).
- 480px for ConfirmUserModal (matches BaseModal default for
  centered modals).

## 4. Behavior preserved

- **ESC closes** ‚Äî comes from BaseModal/BaseDrawer engine. The
  v1 keydown listeners each component installs are removed; the
  engine's listener is the single source.
- **Scrim click closes** ‚Äî same.
- **Focus trap** ‚Äî BaseModal hand-rolled trap (7.1.e ¬ß6). On
  open, focus moves to the first focusable inside the drawer;
  on close, focus restores to the previously-focused element
  (typically the table row that triggered the drawer).
- **Body scroll lock** ‚Äî module-scoped counter from 7.1.e ¬ß6.
  Sequential opens (user drawer ‚Üí wallet drawer) don't leak
  the lock.
- **Animation** ‚Äî BaseDrawer slide-in 220ms ease-out per 7.1.e ¬ß3.
  The v1 `transition: transform 0.22s cubic-bezier(0.4, 0,
  0.2, 1)` is replaced by the engine's transition; visual
  output is equivalent within token tolerance.
- **`v-model:open` vs `:open` + `@close`**: BaseModal/Drawer
  prefer `v-model:open`. The four v1 components use `:open` +
  `@close`. The migration keeps the consumer-facing API as
  `:open` + `@close` for backward compatibility with the four
  call-sites (AdminUsers, AdminWallets, ConfirmModal teleport).
  Internally each component bridges:
  ```ts
  <BaseDrawer :open @update:open="(v) => v || emit('close')">
  ```

## 5. Test plan

### Unit tests

No new unit tests on the four migrated components. They are
integration surfaces; the v1 strategy doesn't unit-test them.
BaseModal/Drawer's 7.1.e suite (22 unit cases + 7 snapshots)
already covers the engine.

### Manual validation (PR description checklist)

For each of the four overlays:

- Opens via its existing trigger (row click on AdminUsers,
  row click on AdminWallets, confirm action from drawer, etc.).
- ESC closes.
- Scrim click closes.
- Tab cycles within the drawer (focus trap).
- Background page does not scroll while drawer is open.
- Sequential open (UserDrawer ‚Üí WalletDrawer back-to-back)
  works; scroll lock releases correctly.
- Drawer internals render identically to v1 (sections, tabs,
  action buttons, fields).
- Light + dark theme.
- Pre/post screenshot pair per drawer (4 overlays √ó 2 themes =
  8 screenshots).
