# Release v0.4.0

## Summary

Phase H1 hardening release — turns the API client into a robust layer with a typed error envelope, debounces the most expensive UI action (chain validation), enforces a 255-character cap on transaction inputs to match the new backend contract, derives the WebSocket URL from `window.location` at runtime, and lands two full sprints of unit tests across the API/composables/organisms layers. Notes written retroactively (2026-05-07).

## Included Changes

- **API error envelope.**
  - New `BlockchainApiError` class plus an `ApiErrorCode` shared union (`RATE_LIMITED`, `BAD_REQUEST`, `NOT_FOUND`, `INTERNAL_ERROR`, `NETWORK_ERROR`, `TIMEOUT`).
  - Axios response interceptor maps 429 / 4xx / 5xx / network / timeout failures into `BlockchainApiError` instances so callers always see one type. Unit tests cover every branch.
- **Validation UX.**
  - Validate-Chain button is debounced so a stuck mouse cannot stampede the backend.
- **Input contract alignment.**
  - Frontend enforces the new BR-TX-05 backend rule: 255-char cap on `sender` and `receiver`, validated client-side before the request leaves the browser.
- **WebSocket URL.**
  - WS URL is now derived from `window.location.protocol` + `host` at runtime (previously hard-coded). Production deployments behind HTTPS automatically upgrade to `wss://` without a rebuild.
- **Test coverage — Sprint 2.**
  - Unit tests for `client` (Axios + interceptor), `usePolling` composable, `useBlockchainWs` composable.
- **Test coverage — Sprint 3.**
  - Unit tests for `MineButton`, `MempoolTable`, and `NodePanel` organisms covering the user-facing flows.

## Validation

- Vitest suite green at the time of cut.
- Coverage thresholds satisfied across the API, composables, and organism layers.
