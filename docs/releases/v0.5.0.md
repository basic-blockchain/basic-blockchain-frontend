# Release v0.5.0

## Summary

Closes the persistence loop on the **Mempool / Transaction History** panel. Until v0.4.0 the confirmed-history table was populated only from live `block_mined` WebSocket events, so a page reload, a hard refresh, or a cache clear emptied it even though the backend had every record persisted in PostgreSQL. The store now hydrates from `GET /api/v1/transactions` (new in `basic-blockchain-simulator@v0.9.0`) on mount, while continuing to receive live appends through the WebSocket.

This release also lands an unrelated lint cleanup that had been carried as pre-existing technical debt: `tests/unit/api/client.test.ts` had two unused imports (`vi`, `axios`) that kept `eslint --max-warnings 0` red. That is now fixed in its own commit.

## Included Changes

- **Mempool view — confirmed history hydration:**
  - `MempoolView.onMounted` now calls both `mempoolStore.fetchPending()` and `confirmedStore.fetchConfirmed()`, so the Transaction History panel is populated immediately on first paint, regardless of whether a `block_mined` event has been observed yet.
  - `useConfirmedTransactionsStore` gains `fetchConfirmed()`, `loading`, and `error` state alongside the existing `addFromBlock` (live append) and `clear` actions.
  - New `getConfirmed()` in `src/api/mempool.ts` calls the new backend endpoint and maps the snake_case payload (`block_index`, `block_timestamp`) onto the existing camelCase `ConfirmedTransaction` shape exposed by `src/domain/transaction.ts`.
- **Lint cleanup:** drop unused `vi` and `axios` imports in `tests/unit/api/client.test.ts`. `MockAdapter` is constructed directly from `@/api/client`, so neither import was needed; `eslint --max-warnings 0` now passes.
- **Documentation refresh:**
  - README rewritten to reflect the actual stack (Vue 3.5 + Vite 6 + TS 5 + Pinia + PrimeVue 4 + Chart.js) and the live application surface (six routes including the rebuilt Mempool view).
  - CLAUDE.md no longer claims the framework is undecided.
  - `package.json` and `package-lock.json` bumped to `0.5.0`.

## Validation

- `npm test -- --run` — 86 unit tests pass across 17 files; new coverage on the success and failure paths of `fetchConfirmed`.
- `npm run lint` — clean (was failing with two `no-unused-vars` errors before this release).
- Manual: with the backend at v0.9.0, mining one block, navigating away from `/mempool`, hitting Ctrl-Shift-R, and returning to `/mempool` keeps the Transaction History populated.

## Notes

- This release depends on `basic-blockchain-simulator@v0.9.0`. Against an older backend, `fetchConfirmed()` will set `error` and the table will appear empty until a new `block_mined` event arrives — non-breaking, no exception bubbles to the user.
- No new Pinia stores were introduced. The contract of `useConfirmedTransactionsStore` is now `{ records, total, loading, error, addFromBlock, fetchConfirmed, clear }`.
- No theme, layout, or routing changes. ADR-001 / ADR-002 / ADR-003 remain in force.
- Next planned work continues on the UX-validation backlog described in [docs/ux-validation-requirements.md](../ux-validation-requirements.md).
