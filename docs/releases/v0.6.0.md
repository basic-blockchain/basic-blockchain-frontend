# Release v0.6.0

## Summary

Phase H+ frontend half (paired with `basic-blockchain-simulator@v0.10.0`). The backend now ships every `Block` payload with a Merkle root and the list of transactions it confirmed; this release adopts that shape end-to-end and surfaces both fields on the chain card. The previous typecheck fix ([#65](https://github.com/basic-blockchain/basic-blockchain-frontend/pull/65)) had to delete `transactions: []` from `Block` test literals because the field did not exist on the interface yet — Phase H+ adds it back, validating the original test author's intuition.

The full plan, locked design decisions, API contract, and Track A / Track B work-item tables are documented in [`docs/phases/phase-h-plus.md`](../phases/phase-h-plus.md).

## Included Changes

- **Domain.** `Block` interface gains `merkleRoot: string` and `transactions: BlockTransaction[]`. `ApiBlock` (snake_case payload) mirrors with `merkle_root` and `transactions`. `blockFromApi` maps the new fields and defaults `transactions` to `[]` when absent so legacy mocks do not break.
- **API parsers.** `mining.ts`, `chain.ts`, and `websocket.ts` keep relying on `blockFromApi`, which now propagates the new fields. `MineResponse` no longer redeclares `transactions` since `ApiBlock` already carries it.
- **Stores.** `chain` store is generic over `Block` and accepts the expanded shape unchanged. `confirmedTransactions` store gains a header comment noting that v0.10.0 nests confirmed transactions on every block (`block.transactions`), making the flat `/api/v1/transactions` endpoint a denormalised view rather than the source of truth — kept for sorting and pagination convenience.
- **UI.** `BlockCard` adds a second `HashChip` row labelled "merkle" rendering `block.merkleRoot`, plus a small "N tx(s)" badge derived from `block.transactions.length`. The previous-hash row gains a "prev" label for symmetry. New CSS rules keep the layout compact on the chain list view.
- **Tests.** `tests/unit/domain/block.test.ts` covers the new `blockFromApi` paths (Merkle mapping, nested tx pass-through, missing-field default). All five existing test files that built `Block` / `ApiBlock` literals (`websocket.test.ts`, `chain.test.ts`, `MineButton.test.ts`, `useBlockchainWs.test.ts`, plus the new block test) now include the v0.10.0 fields so vue-tsc accepts them.
- **Docs.** README badge v0.5.0 → v0.6.0; tests count 86 → 88. Application-surface table notes the new Merkle / tx-count UI. `docs/index.md` adds the release entry. `package.json` and `package-lock.json` bumped to 0.6.0.

## Validation

- `npm run typecheck` — clean (was failing on five TS2739 errors before this PR).
- `npm run lint` — clean.
- `npm test -- --run` — 88/88 pass across 17 files. Coverage holds above the project threshold.

## Migration / breaking notes

- **Pairs with simulator v0.10.0.** Against an older backend, blocks arrive without `merkle_root` and `transactions`; `blockFromApi` defaults `transactions` to `[]` and the merkle row in the block card is hidden via `v-if="block.merkleRoot"`. UI degrades gracefully.
- No breaking change to public composables. Store contracts unchanged: `useChainStore`, `useConfirmedTransactionsStore`, and the WebSocket composable accept the wider Block shape transparently.
- The optional `BlockTransaction` interface lives next to `Block` rather than reusing `Transaction` from `domain/transaction.ts`; the latter still belongs to the mempool form. Consumers needing the on-chain shape should import `BlockTransaction`.

## Notes

- Surfacing the Merkle root in the UI is a deliberate hint that chain integrity now covers transactions. A follow-up phase could add an explicit "Verify integrity" button on the chain view that calls `GET /api/v1/valid` and renders the result alongside each block.
- The flat `confirmedTransactions` store is now technically redundant with chain-derived data. Consolidating it into a computed projection over `useChainStore` is tracked in the deferred backlog (does not need to ship in v0.6.0).
